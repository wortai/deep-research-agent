1. Coding — REST API Design Best Practices

1.1 Overview

RESTful API design establishes a standardized, predictable interface for client-server interactions, emphasizing resource-oriented architecture over RPC-style endpoints. Core best practices revolve around resource naming conventions using nouns in URL paths, precise HTTP method semantics to denote actions, URL path-based versioning for managing changes, consistent error handling with structured responses and standard HTTP status codes, and efficient pagination patterns to handle large datasets. These principles ensure scalability, cacheability, and developer experience, drawing from established guides like Google's API Design Guide and Microsoft's REST API Guidelines. This chapter provides actionable examples, tables, and code snippets for immediate implementation, highlighting common pitfalls such as verb-heavy URLs and deep nesting.

By adhering to these practices, APIs become self-documenting, with HTTP methods carrying the intent (e.g., GET for safe reads, POST for creation), plural nouns for collections, and nested paths limited to two levels for sub-resources. Versioning prevents breaking changes, error responses include machine-readable codes and human-friendly details, and pagination favors cursor-based over offset for performance at scale. Every recommendation includes exact request/response formats, status code mappings, and rationale for trade-offs like idempotency.

1.2 Resource Naming Conventions and HTTP Methods

REST APIs must represent resources as nouns in URL paths, with HTTP methods specifying the action. This separation promotes uniformity: the path identifies the resource (e.g., /users for the collection of user resources), while the method defines the operation. Collections use plural nouns like /users or /orders, single resources incorporate IDs like /users/123, and relationships employ shallow nesting like /users/123/orders. Avoid nesting beyond two levels to prevent unwieldy paths; for deeper queries, append query parameters such as ?include=product,category.

Method Path Use Case
GET /users List all users
GET /users/123 Get user 123
POST /users Create a new user
PUT /users/123 Replace user 123 entirely
PATCH /users/123 Partially update user 123
DELETE /users/123 Delete user 123

The table above illustrates correct resource-oriented paths. For nested resources belonging to a parent, use paths like GET /users/123/orders to list orders for user 123, GET /users/123/orders/456 for a specific order, or POST /users/123/orders to create an order under that user. This maintains clarity without excessive depth.Source: Google API Design GuideSource: Microsoft REST API Guidelines

Common Mistake: Verbs in URLs

Avoid verb-laden paths, as they duplicate HTTP method semantics and lead to inconsistent APIs:

  • GET /getUsers
  • POST /createUser
  • POST /updateUser/123

These violate REST principles by embedding actions in paths rather than using methods.

HTTP methods carry precise semantics regarding safety (can be repeated without side effects), idempotency (multiple identical requests yield the same result), and body usage. GET is safe and idempotent for reads without a body; POST is neither but creates resources; PUT fully replaces idempotently; PATCH partially updates (not strictly idempotent per spec but implemented as such in practice); DELETE idempotently removes with optional body.

Method Idempotent Safe Body? Use Case
GET Yes Yes No Read/retrieve resource
POST No No Yes Create new resource
PUT Yes No Yes Full replacement
PATCH No* No Yes Partial update
DELETE Yes No Optional Remove resource
*PATCH is technically not idempotent by spec, but should be implemented idempotently in practice.

This table details method properties, enabling clients to retry safely (e.g., idempotent PUT/DELETE) and servers to cache GET responses. Consider a complete POST /api/v1/users request and 201 Created response, including self-links for HATEOAS navigation and related sub-resources.

POST /api/v1/users
Content-Type: application/json
{
  "name": "Jane Smith",
  "email": "jane@example.com",
  "role": "admin"
}
Response — 201 Created
{
  "id": "usr_abc123",
  "name": "Jane Smith",
  "email": "jane@example.com",
  "role": "admin",
  "createdAt": "2024-01-15T10:30:00Z",
  "_links": {
    "self": { "href": "/api/v1/users/usr_abc123" },
    "orders": { "href": "/api/v1/users/usr_abc123/orders" }
  }
}

The response echoes the input, adds server-generated fields like id and createdAt, and provides _links for discoverability, aligning with hypermedia principles.

1.3 API Versioning Strategies

Versioning manages evolutionary changes without breaking clients. For major/breaking changes, increment the version (v1 to v2); for additive changes like new optional fields or endpoints, evolve within the version. Three primary strategies exist, each balancing simplicity, cacheability, and discoverability.

Strategy Example Pros Cons
URL Path (Most Common) GET /api/v1/users
GET /api/v2/users		 Simple, explicit, cacheable Clutters URLs, version proliferation
Header GET /api/users
Header: API-Version: 2024-01-15		 Clean URLs, date-based Less discoverable, browser testing harder
Query Parameter GET /api/users?version=2 Easy to test Can be ignored by clients

URL path versioning is recommended for its explicitness and caching benefits, as proxies can differentiate /v1 vs /v2. Date-based headers suit micro-versioning, while query params offer flexibility but risk oversight.Source: Stripe API Reference

1.4 Error Handling

Consistent error responses across endpoints include a structured JSON object with code, message, field-specific details, requestId for tracing, and timestamp. Pair with precise HTTP status codes to signal request lifecycle issues.

Status Meaning When to Use
200 OK Successful GET, PUT, PATCH
201 Created Successful POST (new resource created)
204 No Content Successful DELETE
400 Bad Request Invalid request body, missing required fields
401 Unauthorized Missing or invalid authentication token
403 Forbidden Authenticated but no permission
404 Not Found Resource doesn't exist
409 Conflict Duplicate resource, version conflict
422 Unprocessable Valid JSON but semantic errors (validation)
429 Too Many Requests Rate limit exceeded
500 Internal Error Unexpected server error

Note 422 for validation failures despite syntactically valid JSON, distinguishing from 400. A sample validation error (e.g., 422) provides granular details for client correction.

422 Unprocessable Entity Error Response
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "The request body contains invalid fields",
    "details": [
      {
        "field": "email",
        "message": "Must be a valid email address",
        "code": "INVALID_FORMAT"
      },
      {
        "field": "age",
        "message": "Must be between 18 and 120",
        "code": "OUT_OF_RANGE"
      }
    ],
    "requestId": "req_xyz789",
    "timestamp": "2024-01-15T10:30:00Z"
  }
}

This format enables automated handling (via code/details) and debugging (requestId/timestamp).Source: GitHub REST API

1.5 Pagination Patterns

Pagination prevents overwhelming responses from large collections. Offset-based is simple but scans entire datasets inefficiently; cursor-based uses opaque tokens for efficient resumption; keyset leverages indexed columns for optimal performance without offsets.

GET /api/v1/users?limit=20&offset=40 (Offset-Based)
{
  "data": [...],
  "pagination": {
    "total": 1247,
    "limit": 20,
    "offset": 40,
    "hasMore": true
  }
}

Offset scales poorly for millions of records due to full scans.

GET /api/v1/users?limit=20&cursor=eyJpZCI6MTIzfQ== (Cursor-Based, Recommended)
{
  "data": [...],
  "pagination": {
    "nextCursor": "eyJpZCI6MTQzfQ==",
    "hasMore": true
  }
}

Cursors encode position (e.g., base64({id:123})), enabling server-side efficiency without exposing internals.

Keyset pagination uses queries like GET /api/v1/users?limit=20&after_id=usr_123&sort=created_at, scanning ranges on indexes for constant-time performance.

Always include rate-limiting headers on responses: X-RateLimit-Limit: 1000, X-RateLimit-Remaining: 847, X-RateLimit-Reset: 1705312200 (Unix timestamp). Cursor-based is preferred for large-scale APIs like user lists.Source: Stripe API ReferenceSource: GitHub REST API
