VISUAL MOOD  The report embodies the pristine, highly functional aesthetic of top-tier developer documentation like Stripe Docs. It feels technical, clinical, and precise. A stark contrast is maintained between the crisp white reading surface and deep dark-mode code blocks, reinforcing the boundary between theory and execution. Monospace fonts, tight grid-aligned tables, and crisp UI-like borders project engineering rigor.    5. REPORT INTENT & PRESENTATION  This report serves as an authoritative best-practices guide for RESTful API design. The primary goal is to provide developers and architects with actionable, copy-paste-ready standards for endpoints, versioning, error handling, and pagination. Visual presentation is structural, relying heavily on syntax-highlighted code blocks, explicit API endpoint tables, and sequence diagrams. Information density is prioritized in reference sections, while prose is punchy and focused on the 'why' behind architectural trade-offs.    6. VISUAL DECISION TREE  * IF demonstrating an API request or response payload THEN use a dark-mode code block with a header specifying the HTTP method/endpoint and language tag.  * IF establishing resource naming conventions or status codes THEN use a structured table (Endpoint | Method | Use Case | Status).  * IF explaining data flow, sequence of requests, or pagination architecture (e.g. offset vs cursor) THEN use an SVG flow or sequence diagram.  * IF outlining critical architectural mistakes or deprecations (e.g. verbs in URLs) THEN use a Warning/Danger callout box with an orange or red border.  * IF comparing performance or scaling implications of different design patterns THEN use a Charts.css column chart.  * IF showing a specific function signature or parameter data type THEN use monospace text inline or a dense parameter table.    7. SVG & CHARTS.CSS CONSTRUCTION  a) SVG Construction: Standard viewBox='0 0 800 400' for sequence diagrams, viewBox='0 0 400 400' for smaller components. Font families: Inter for diagram titles and node labels, 'Fira Code' for payload representations. Stroke Widths: 1.5px for reference lines, 2px for primary connections/arrows, 2.5px for component borders. Colors: Use only defined palette (Components #EBF8FF with #3182CE borders; Lines #2D3748). Accessibility: Every SVG must have <title> and <desc>. Domain-specific patterns: Flowcharts use rounded rectangles (rx='4'), sequence diagrams use vertical lifelines with horizontal activation vectors.  b) Charts.css Construction: Use column/line charts for performance/latency comparisons; Horizontal bar charts for categorical metrics. Required modifier classes: show-labels, show-headings, show-data. Color sequence: #2B6CB0, #319795, #805AD5. Use show-primary for recommended approaches. Do NOT use Charts.css if you have fewer than 3 data points.    8. WHAT TO AVOID  * Never use serif fonts or center-aligned prose; it ruins the developer documentation feel.  * Never use monochrome code blocks. All code MUST be syntax-highlighted using inline spans, and every block MUST have a top header specifying context.  * Never write pseudocode. REST examples must use valid JSON and exact HTTP headers.  * Never use complex charts when a simple markdown table of endpoints provides faster scannability.  * Do not leave SVG diagrams unannotated. Every arrow representing an API call must indicate the payload or HTTP method.
