Choose the correct stream_mode and use Annotated reducers in your state schema to control whether you get deltas or full state objects.

LangGraph provides four primary streaming modes. How your state is updated depends on whether you've defined a reducer function (like operator.add) for your state keys.

from typing import Annotated
from operator import add
from typing_extensions import TypedDict
from langgraph.graph import StateGraph

# 1. State with a Reducer (streams deltas in "updates" mode)
class State(TypedDict):
    # Annotated with 'add' means new values are APPENDED, not replaced
    messages: Annotated[list, add] 
    # No reducer means the value is REPLACED
    status: str

builder = StateGraph(State)
# ... add nodes and compile

Supported Streaming Modes
Each mode handles state updates and data transmission differently:

updates: Streams only the changes (deltas) from each node. If a key has a reducer (like messages), you only get the new messages. If it doesn't (like status), you get the whole new value for that key.
values: Streams the entire state after every node. This is the "send as whole" mode.
messages: Streams LLM tokens and metadata in real-time as they are generated.
custom: Streams arbitrary data you emit manually using StreamWriter inside your nodes or tools.
When Data "Sends as Whole" (Instead of Streaming)
There are three common scenarios where you will receive the entire data object instead of a stream:

Using stream_mode="values": This mode is designed to return the full state at every step. If you want deltas, you must switch to updates.
Missing Reducers: If your state key is just a list (without Annotated[list, add]), LangGraph replaces the old list with the new one. In updates mode, the "delta" for a replacement is the entire new list.
Model Configuration: If your LLM is initialized with streaming=False or doesn't support streaming, stream_mode="messages" will only return the final AIMessage once it is fully complete.
# Example of streaming multiple modes simultaneously
async for mode, chunk in graph.astream(
    {"messages": [HumanMessage(content="Hello")]}, 
    stream_mode=["updates", "messages"]
):
    if mode == "updates":
        # Only get what the last node returned
        print(f"Node Update: {chunk}")
    elif mode == "messages":
        # Get individual tokens from the LLM
        token, metadata = chunk
        print(token.content, end="|", flush=True)

To stream custom progress (e.g., "Scanning database..."), use the StreamWriter (or config.writer in JS) and ensure you include stream_mode="custom" in your stream call.

stream_mode="messages" is the only mode that sends data token-by-token directly from the model before the state is even updated.

You are correct that nodes typically return a dictionary to update the graph state all at once. However, stream_mode="messages" works differently by "hooking" into the LLM calls inside your nodes. It emits tokens as they are generated by the model, bypassing the usual "wait for node to finish" cycle.

How "messages" Mode Bypasses State
When you use stream_mode="messages", LangGraph captures the internal stream of the chat model.

The State: Only updates after the node finishes. The final, complete message is then added to your MessagesState or state dictionary.
The Stream: Emits (message_chunk, metadata) tuples in real-time. These chunks are not pulled from the state; they are pulled directly from the model's output stream.
# Even though 'call_model' returns a dict at the end, 
# "messages" mode streams tokens WHILE 'call_model' is running.
async for message, metadata in graph.astream(
    {"messages": [HumanMessage(content="Explain quantum physics")]},
    stream_mode="messages"
):
    # 'message' is an AIMessageChunk (one or more tokens)
    print(message.content, end="|", flush=True)

Why "updates" and "values" Send Whole Objects
In contrast, updates and values are tied strictly to the Graph State:

updates: Waits for a node to return its dictionary (e.g., return {"messages": [ai_msg]}). Once returned, it emits that specific dictionary.
values: Waits for a node to return, applies the update to the state, and then emits the entire new state.
If you want the "typewriter" effect in your UI, you must use stream_mode="messages". If you only care about which node just finished and what it changed, use stream_mode="updates".

updates mode sends the whole dictionary returned by your node, not individual tokens.

It does not take the state and break it into tokens. Instead, it waits for a node to finish its entire execution and then sends whatever that node returned to update the state.

Comparison of Modes
Feature	updates Mode	messages Mode
Unit of Data	Dictionary (State Delta)	Message Chunk (Tokens)
Timing	Sent AFTER the node finishes	Sent DURING node execution
Source	The node's return value	Direct from the LLM stream
Use Case	Tracking which node ran and what it changed	Real-time typewriter effect in UI
Example: Updates vs Messages
If your node calls an LLM to generate a long paragraph:

In updates mode: You see nothing for 10 seconds. Then, once the model finishes, you receive the whole paragraph at once as a single dictionary: {'node_name': {'messages': [AIMessage(content="...")]}}.
In messages mode: You receive 50+ small chunks of data (tokens) over those 10 seconds. Each chunk contains only 1-2 words.
# 'updates' will give you the full object at the end of the node
async for chunk in graph.astream(input, stream_mode="updates"):
    print(chunk) 
    # Output: {'agent': {'messages': [AIMessage(content="The capital of France is Paris.")]}}

# 'messages' will give you chunks as they are generated
async for chunk in graph.astream(input, stream_mode="messages"):
    print(chunk[0].content)
    # Output: The| capital| of| France| is| Paris|.
Copy
If you want both—to know when nodes finish AND to see tokens in real-time—you can pass a list: stream_mode=["updates", "messages"].
